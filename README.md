# Spot the Fake Photo — Screen Recapture Detector

Detects whether an image is a **real photo** or a **photo of a screen** (someone
re-photographing a phone/laptop/printout instead of the real thing).

Approach: **classical signal processing**, not a trained neural net. Eight
hand-engineered features (FFT/moiré analysis, glare, blur uniformity, texture
periodicity, color cast, saturation, edge density) feed a small logistic
regression. No GPU, no large model weights.

---

## Files — what each one does

| File | Purpose |
|---|---|
| `features.py` | Extracts the 8 signal-processing features from an image (FFT moiré analysis, highlight/glare ratio, sharpness uniformity, LBP texture periodicity, color cast, saturation, edge density). This is the actual "detector" — all the real logic lives here. |
| `model.py` | Tiny logistic regression: `sigmoid(w · standardize(features) + b)`. Loads `model.json` if it exists (trained), otherwise falls back to sane hand-set default weights so the project runs out of the box even with zero data. |
| `predict.py` | **The required CLI entry point.** `python predict.py image.jpg` prints a score 0–1. Also supports `--time` flag to print latency + per-feature breakdown (useful for debugging/demo). |
| `train.py` | Reads your `real/` and `screen/` folders, extracts features for every image, fits a logistic regression (no sklearn needed — small hand-written gradient descent), 5-fold cross-validates it, picks the best decision threshold, and writes `model.json`. Run this once you have data. |
| `model.json` | **Generated by `train.py`** — your calibrated weights, normalization stats, accuracy, and threshold. `predict.py` automatically uses it once it exists. Not present until you train. |
| `server.py` | Tiny Flask backend that exposes `/analyze` (POST an image, get back JSON score + features) so the browser demo can call your Python detector live. Also serves `demo.html` at `/`. |
| `demo.html` | Live camera/upload web demo — capture a photo or drag-drop one, see the score, the per-feature breakdown bars, and a plain-English explanation of *why* it scored that way. Talks to `server.py`. |
| `download_dataset.py` | Optional helper to bulk-download images from a list of direct URLs into `real/`/`screen/`. Not needed if you just drag files into the folders yourself (recommended — simpler). |
| `requirements.txt` | Python dependencies. |
| `real/`, `screen/` | Empty folders — put your training photos here before running `train.py`. |

## How to run this in VS Code

### 1. Open the project
Open the `project/` folder in VS Code (`File → Open Folder…`).

### 2. Create a virtual environment (recommended)
Open the VS Code integrated terminal (`` Ctrl+` ``) and run:

```bash
python -m venv venv
```

Activate it:
- **Windows:** `venv\Scripts\activate`
- **Mac/Linux:** `source venv/bin/activate`

VS Code may prompt "Select environment" — pick the `venv` interpreter (bottom-right corner or `Ctrl+Shift+P → Python: Select Interpreter`).

### 3. Install dependencies

```bash
pip install -r requirements.txt
```

### 4. Get your dataset into `real/` and `screen/`

Since you have Google Drive links: **download the files from Drive to your
computer first** (Drive's "Download" button, or download-all-as-zip for a
folder), then drag the image files straight into the `real/` and `screen/`
folders in VS Code's file explorer. This is far more reliable than trying to
script around Drive's share-link redirects.

You want roughly:
- `real/` → ~50 photos of real objects/scenes
- `screen/` → ~50 photos of a screen or printout showing a picture

Any common format works, including iPhone's HEIC: `.jpg`, `.jpeg`, `.png`,
`.webp`, `.bmp`, `.heic`, `.heif`. The `pillow-heif` package in
`requirements.txt` handles HEIC decoding automatically — just drop the files
in as-is, no manual conversion needed.

*(If you ever do have direct, public image URLs — not Drive share links — you
can instead list them in `real_urls.txt` / `screen_urls.txt` and run
`python download_dataset.py`, but for Drive links, manual download is simpler.)*

### 5. Run predict.py with the un-trained defaults (sanity check)

Before training, the script already works using hand-set default weights:

```bash
python predict.py real/your_image.jpg
```

This should print a number close to 0. This step just confirms your
environment/dependencies are correctly installed.

### 6. Train the model on your data

```bash
python train.py --real real/ --screen screen/
```

This prints:
- 5-fold cross-validation accuracy (your honest, unbiased estimate — **this is
  the number to put in your write-up**, not raw training accuracy)
- Final fitted feature weights (which signals mattered most)
- Writes `model.json`

### 7. Re-run predict.py — now using the trained model

```bash
python predict.py real/your_image.jpg
python predict.py screen/your_image.jpg --time
```

The `--time` flag also prints latency in ms and the full feature breakdown.

### 8. Run the live camera demo

```bash
python server.py
```

Then open `http://localhost:5000` in your browser. You can use your webcam or
upload images and see the score update live with the feature breakdown and a
plain-English explanation.

## What to put in your submission note (per the brief)

1. **Approach** — classical signal processing (FFT/moiré, glare, blur
   uniformity, texture periodicity, color, edges) + a tiny logistic regression
   on top. No deep model.
2. **Accuracy** — report the **cross-validation accuracy** number `train.py`
   prints, with an honest caveat about your dataset size (~100 images is small;
   say so).
3. **What you'd improve** — more data, more screen/lighting variety, maybe add
   a chromatic-aberration or JPEG-artifact feature, calibrate threshold more
   carefully with more held-out data.
4. **Required numbers:**
   - **Latency** — run `python predict.py image.jpg --time` and report the
     `latency` line (should be roughly 15–40 ms on a laptop CPU).
   - **Cost per image** — this runs entirely on-device/CPU with no GPU and no
     API calls, so cost is effectively **$0 per image** if run client-side; if
     hosted on a cheap cloud CPU instance, latency this low means you could
     serve well over 1M images per dollar of compute — state your assumption
     explicitly (e.g. "$0.05/hr CPU instance, ~25ms/image → ~144,000
     images/hour → ~$0.0003 per 1,000 images").
